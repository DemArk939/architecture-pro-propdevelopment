## Список требований:

1. Необходимо классифицировать данные.
   Данные, связанные с пользователями и способные его идентифицировать (биометри, адреса, номера автомобилей и тд),
   должны подлежать строгому контролю.
2. Необходимо производить шифрование данных.
   Все данные, биометрия, адреса, номера автомобилей и связанные персональные денные должны храниться в зашифровонном
   виде.
   Должны использоваться алгоритмы шифрования и TLS для защиты данных при передаче между клиентом, API и внешними
   сервисами.
   Ключи шифрования должны хранятся в защищённых аппаратных модулях (HSM).
3. Данные должны быть обезличены.
   Хранение биометрии, адресов и номеров автомобилей должны храниться без прямой привязки к другой идентифицирующей
   информаци.
   Для передачи в средства хранения и для аналитических целей, данные пользователя должны обезличиваться.
4. Доступ к данным должен иметь только ограниченный круг сотрудников.
   Каждый сотрудник получает доступ на основе принципа наименьших привилегий (Least Privilege).
5. Все операции с данными должны фиксироваться в журналах. Журналы также должны шифроваться.
6. Использование стандарта OAuth 2.0 / OpenID Connect для аутентификации и доступа к API мобильного приложения.
7. При интеграции с внешними сервисами необходимо передавать только минимально необходимую информацию.
   Для того чтобы открыть шлагбаум, необходимо передать во внешнюю систему номер автомобиля и адрес.
   Для того чтобы открыть домофон, необходимо передать во внешнюю систему информацию о пользователе и адрес.
   Так как при передаче используется чувствительная информация, то все запросы к API используют https и TLS, проверку
   аутентификации и авторизации, например OAuth 2.0.
8. Краткоживущие access token’ы, привязанные к конкретному пользователю и его объектам недвижимости
   (RBAC/ABAC: пользователь может открыть только «свои» домофоны/шлагбаумы)

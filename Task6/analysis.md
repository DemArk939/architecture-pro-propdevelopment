# Отчёт по результатам анализа Kubernetes Audit Log

## Подозрительные события

1. Доступ к секретам:
    - Кто: minikube-user
    - Где: "namespace": "kube-system"
    - Почему подозрительно: пользователь напрямую запрашивает все секреты системного namespace, что может быть опасно с точки зрения безопасности
    - 
    - Кто: minikube-user
    - Где: "namespace": "kube-system"
    - Почему подозрительно: пытался выдать себя за system:serviceaccount:secure.

2. Привилегированные поды:
    - Кто: system:node:minikube и system:serviceaccount:kube-system:calico-cni-plugin
    - Комментарий: Обнаружен GET-запрос к привилегированному поду с командой sleep 3600.
      Привилегированные поды дают root-доступ к системе. Необходимо проверить, кто создал этот под и зачем.

3. Использование kubectl exec в чужом поде:
    - Кто: Не обнаружено в логе

4. Создание RoleBinding с правами cluster-admin:
    - Кто: Не обнаружено в логе
    - К чему привело: ...

5. Удаление audit-policy.yaml:
    - Команда kubectl delete -f работает только с манифестами Kubernetes объектов (Pod, ConfigMap, Deployment и т.д.),
      а не с файлами хост-системы.
      При выполнении команды получаем:
      Результат: Error: 
      Событий аудита НЕТ

## Вывод
Политика позволяет создавать pods с privileged: true, что может дать root-доступ к узлу.
Администраторы (system:masters) имеют неограниченный доступ.
Работа кластера может быть полностью нарушена
